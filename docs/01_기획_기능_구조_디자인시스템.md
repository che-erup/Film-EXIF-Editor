> 작성일: 2026-06-28

## 0. 먼저, 이 프로그램이 왜 필요한가

디지털 카메라는 셔터를 누르는 순간 촬영 날짜, 카메라/렌즈 모델, 조리개, 셔터스피드, ISO 같은 정보를 사진 파일 안에 **EXIF**라는 형태로 자동으로 기록합니다.

그런데 필름 카메라는 이 EXIF를 전혀 남기지 않습니다. 필름으로 찍은 뒤 스캔을 하면, 파일 안에 들어가는 정보는 "스캐너가 언제 스캔했는가"뿐입니다. 그 결과 두 가지 문제가 생깁니다.

- **촬영 날짜가 엉킵니다.** 1995년에 찍은 사진도 "2026년 촬영"으로 표시됩니다. 스캔한 날짜가 곧 촬영 날짜로 들어가기 때문입니다.
- **카메라·렌즈·필름 정보가 비어 있습니다.** 어떤 바디로, 어떤 렌즈로, 어떤 필름으로 찍었는지 사진 파일만 봐서는 알 수 없습니다.

이 툴은 사용자가 스캔 사진을 불러와서 이 빈칸들을 **쉽고 빠르게, 여러 장을 한 번에** 채워 넣도록 돕는 프로그램입니다.

> 참고: 필름 스캔본은 본질적으로 "메타데이터가 비어 있는 파일"이라는 점이 여러 필름 사진가들의 공통된 출발점입니다. ([Emory Dunn](https://emorydunn.com/blog/2023/12/film-metadata/), [shom.dev](https://shom.dev/posts/20250928_exif-for-film-photos/))

## 1. 어떤 정보를 수정하나 — EXIF 필드 지도

프로그램이 다룰 정보를 "사용자가 입력하는 항목 → 실제 파일에 저장되는 EXIF 태그 이름"으로 정리했습니다. 이 표가 앱의 입력 화면(폼) 구조를 그대로 결정합니다.

정보는 성격에 따라 **롤 공통(한 통 전체가 동일)**과 **컷별(사진마다 다름)** 두 층으로 나눕니다. 이 구분이 곧 입력 방식(일괄 적용 vs 개별 수정)을 결정합니다.

### 롤 공통 정보 — 한 번 입력해 전체 컷에 적용

| 사용자가 보는 입력칸 | 실제 저장되는 EXIF 태그 | 메모 |
| --- | --- | --- |
| 카메라 제조사 | `Make` | 예: Nikon, Canon, Leica |
| 카메라 모델(바디) | `Model` | 예: Nikon FM2, Leica M6 |
| 렌즈 제조사 | `LensMake` | 예: Nikon, Voigtländer |
| 렌즈 모델 | `LensModel` | 예: 50mm f/1.4 |
| 필름 종류 | (표준 태그 없음) → `UserComment`(+XMP) | 아래 ※ 참고 |
| 현상소 | (표준 태그 없음) → `UserComment`(+XMP) | 화면엔 전용칸, 저장 시 UserComment에 합쳐 기록 |

### 컷별 정보 — 사진마다(또는 구간마다) 따로 수정

| 사용자가 보는 입력칸 | 실제 저장되는 EXIF 태그 | 메모 |
| --- | --- | --- |
| 촬영 날짜·시간 | `DateTimeOriginal` | 가장 중요. 같은 롤이라도 컷마다 다를 수 있음 |
| (보조) 생성/디지털화 날짜 | `CreateDate` / `DateTimeDigitized` | 스캔한 날짜를 따로 보존하고 싶을 때 |

### 확장(2차) 항목

| 사용자가 보는 입력칸 | 실제 저장되는 EXIF 태그 |
| --- | --- |
| 초점거리 | `FocalLength` |
| 조리개 | `FNumber` (`ApertureValue`) |
| 셔터스피드 | `ExposureTime` |
| 필름 감도 / 노출지수 | `ISOSpeedRatings`, `ExposureIndex` |
| 현상 정보(현상소·현상시간) | `UserComment` 등 보조 필드 |
| GPS 촬영 위치 | `GPSLatitude` / `GPSLongitude` |

> ※ **필름 종류에는 정해진 표준 EXIF 태그가 없습니다.** 그래서 필름 사진가용 도구(LensTagger 등)는 필름 제조사·필름명·노출지수·현상 정보를 주로 `UserComment` 필드에 모아 적는 방식을 씁니다. 우리 앱도 이 관례를 따르고, 추가로 XMP에도 함께 기록해 호환성을 높이는 방향을 권장합니다. ([35mmc](https://www.35mmc.com/21/05/2020/bridging-analog-photography-and-digital-metadata-with-exif-notes-by-babak-farshchian/), [shom.dev](https://shom.dev/posts/20250928_exif-for-film-photos/))

## 2. 기능 구상

### 2-1. 핵심 기능 (MVP — 가장 먼저 만들 것)

작업의 중심은 **2단계 워크플로**입니다: ① 롤 공통 정보를 한 번 입력해 전체에 깔고 → ② 사진을 다중 선택해 컷별 정보(주로 날짜)를 수정.

1. **사진 불러오기** — 파일/폴더를 드래그 앤 드롭하거나 선택해서 한 번에 여러 장 불러오기 (JPEG, TIFF, DNG 등)
2. **① 롤 공통 정보 입력 → 전체 적용** — 카메라·렌즈·필름·현상소를 한 번 입력하면 불러온 모든 컷에 동일하게 적용
3. **② 다중 선택 → 컷별 정보 수정** — 썸네일을 여러 장 선택해, 그 묶음에만 촬영 날짜·시간을 따로 지정 (롤 공통 값은 유지)
4. **날짜 입력 보조** — 구간 단위로 다른 날짜를 주거나, 같은 날 연속 컷은 시간을 1초/1분씩 자동 증가시켜 촬영 순서대로 정렬되게 함. 날짜를 모를 땐 연/월만 입력하는 느슨한 입력 허용
5. **저장 전 미리보기** — 어떤 값이 어떻게 바뀌는지 적용 전에 확인
6. **원본 보존(백업)** — 수정 시 원본을 자동 백업하거나 사본에 적용 (실수 복구 안전장치)
7. **변경 저장** — 실제 파일의 EXIF에 기록

### 2-2. 편의 기능 (v2 — 다음 단계)

- **프리셋(조합 저장)**: "Nikon FM2 + 50mm f/1.4 + Portra 400" 같은 자주 쓰는 조합을 저장해 두고 클릭 한 번으로 적용
- **롤 단위 관리**: 필름 한 통을 하나의 묶음으로 보고, 묶음 전체에 공통 정보 + 컷별 개별 정보 관리
- **날짜 일괄 보정**: 모든 사진을 특정 날짜로 설정하거나, 일정 간격으로 자동 증가
- **CSV 가져오기/내보내기**: 촬영 노트를 표로 정리해 한꺼번에 입력

### 2-3. 고급 기능 (v3 — 여력이 될 때)

- 지도에서 클릭해 GPS 위치 지정
- 필름 데이터베이스 내장(주요 필름 목록 자동완성)
- 다국어, 자동 업데이트

### 기능 우선순위 한눈에 보기

| 단계 | 목표 | 핵심 산출물 |
| --- | --- | --- |
| **MVP** | "한 통 분량 사진의 날짜·카메라·렌즈·필름을 한 번에 채운다" | 불러오기 · 편집 폼 · 일괄 적용 · 백업 · 저장 |
| **v2** | "반복 작업을 줄인다" | 프리셋 · 롤 관리 · 날짜 보정 · CSV |
| **v3** | "정보를 풍부하게" | GPS · 필름 DB · 다국어 |

## 3. 프로그램 구조도 (아키텍처)

복잡해 보여도 핵심은 세 덩어리입니다. **사용자가 보고 만지는 화면 / 일을 처리하는 두뇌 / 실제로 파일을 고치는 엔진.**

```
┌─────────────────────────────────────────────────────────────┐
│  ① 화면 (UI) — 사용자가 보고 클릭하는 부분                    │
│  ─────────────────────────────────────────────────────────  │
│   · 사진 목록(썸네일)   · 미리보기   · 메타데이터 입력 폼     │
│   · 일괄 적용 버튼      · 프리셋 선택                          │
└───────────────────────────────┬─────────────────────────────┘
                                 │  (사용자 입력 전달)
                                 ▼
┌─────────────────────────────────────────────────────────────┐
│  ② 두뇌 (앱 로직) — 입력을 정리하고 무엇을 할지 결정           │
│  ─────────────────────────────────────────────────────────  │
│   · 입력값 검증(날짜 형식 등)   · 일괄 적용 대상 계산          │
│   · 프리셋/설정 저장·불러오기   · 백업 규칙                    │
└───────────────────────────────┬─────────────────────────────┘
                                 │  (수정 명령 전달)
                                 ▼
┌─────────────────────────────────────────────────────────────┐
│  ③ EXIF 엔진 — 실제 사진 파일의 메타데이터를 읽고 쓰는 부분    │
│  ─────────────────────────────────────────────────────────  │
│   · ExifTool (검증된 표준 도구) 사용                          │
└───────────────────────────────┬─────────────────────────────┘
                                 │
                                 ▼
                    📁 사용자의 사진 파일들
                    (원본 백업 → 사본/원본에 EXIF 기록)
```

### 데이터가 흐르는 순서 (예시)

1. 사용자가 사진 30장을 끌어다 놓는다 → **①화면**이 썸네일로 보여준다
2. "Portra 400, Nikon FM2, 1998-05-10" 입력 후 30장 모두 선택 → "일괄 적용" 클릭
3. **②두뇌**가 날짜 형식이 올바른지 확인하고, 30장 각각에 적용할 명령을 만든다
4. **③EXIF 엔진**이 원본을 백업한 뒤 각 파일의 EXIF 태그를 실제로 고친다
5. 결과를 **①화면**에 "30장 완료"로 표시한다

### 왜 ExifTool을 엔진으로 쓰나

EXIF를 직접 다루는 코드를 처음부터 만드는 것은 매우 어렵고 위험합니다(파일이 깨질 수 있음). **ExifTool**은 거의 모든 사진 포맷의 메타데이터를 안전하게 읽고 쓰는, 수십 년간 검증된 표준 도구입니다. 우리 앱은 이 ExifTool을 내부 엔진으로 품고, 사용자에게는 쉬운 화면만 보여주는 구조가 안전하고 빠릅니다. ([ExifTool by Phil Harvey](https://exiftool.org/))

### 추천 기술 스택 (비개발자 관점 요약)

| 부분 | 추천 | 이유 |
| --- | --- | --- |
| 앱 껍데기 | **Tauri (v2)** | 용량이 매우 작고(수 MB) 가벼우며 보안이 좋음. 2026년 신규 데스크톱 앱의 기본 권장 |
| 화면(UI) | 웹기술(HTML/CSS/JS) | 디자인 자유도가 높고 자료가 많음 |
| EXIF 처리 | **ExifTool** 내장 | 검증된 표준 엔진 |

대안으로 **Electron**(VS Code, Slack이 쓰는 방식)도 있습니다. 자료와 사례가 더 많아 배우기 편한 대신 앱 용량이 150MB 이상으로 커집니다. "가볍고 깔끔한 도구"가 목표라면 Tauri, "사례 많고 무난하게"가 목표라면 Electron이 맞습니다. ([Tauri vs Electron 2026](https://www.buildmvpfast.com/blog/tauri-v2-vs-electron-desktop-apps-2026), [digitalapplied](https://www.digitalapplied.com/blog/desktop-apps-web-stack-tauri-electron-deno-wails-2026))

## 4. 디자인 시스템

필름 사진의 따뜻하고 아날로그적인 감성을 담되, 작업 도구로서 **정보가 한눈에 보이고 손이 빠른** 화면을 지향합니다.

### 4-1. 화면 레이아웃 — 3분할 구조

대량의 사진을 다루는 정리 도구는 대부분 이 구조를 씁니다(Lightroom 등과 동일한 직관).

```
┌──────────────┬───────────────────────────┬──────────────────────┐
│  왼쪽         │  가운데                    │  오른쪽               │
│  사진 목록    │  큰 미리보기               │  메타데이터 편집 폼   │
│  (썸네일)     │  (선택한 사진)             │                       │
│              │                            │  ┌─ 롤 공통 ───────┐ │
│  □ □ □ □      │      [ 사진 ]              │  │ 카메라  ____    │ │
│  □ □ □ □      │                            │  │ 렌즈    ____    │ │
│  □ □ ▣ □      │                            │  │ 필름    ____    │ │
│  □ □ □ □      │                            │  │ 현상소  ____    │ │
│              │                            │  │ [전체에 적용]   │ │
│  [전체 선택]  │                            │  └─────────────────┘ │
│              │                            │  ┌─ 컷별(선택분) ──┐ │
│              │                            │  │ 촬영일  ____    │ │
│              │                            │  │ [선택 항목 적용] │ │
│              │                            │  └─────────────────┘ │
└──────────────┴───────────────────────────┴──────────────────────┘
 하단 바: 선택 30장 / 전체 36장        [ 프리셋 ▾ ]   [ 저장 ]
```

- **왼쪽**: 불러온 사진들을 썸네일로. 여러 장 다중 선택 → 컷별 수정의 대상.
- **가운데**: 선택한 사진을 크게 보며 확인.
- **오른쪽**: 입력 폼을 **롤 공통 / 컷별** 두 구역으로 분리. 위쪽은 "전체에 적용", 아래쪽은 선택한 사진에만 적용. 이 화면 구조가 2단계 워크플로를 그대로 안내합니다.

### 4-2. 색상 (Color)

필름의 따뜻한 종이/현상 톤을 베이스로, 작업 도구답게 채도는 낮게 잡습니다.

| 용도 | 이름 | 값(예시) | 설명 |
| --- | --- | --- | --- |
| 배경 | Ink (다크) | `#1C1A17` | 사진이 돋보이도록 어두운 따뜻한 차콜 |
| 패널 배경 | Charcoal | `#26231F` | 목록·폼 영역 |
| 기본 텍스트 | Paper | `#EDE6DA` | 따뜻한 화이트(순백 지양) |
| 보조 텍스트 | Muted | `#9A9085` | 라벨·설명 |
| 강조(주요 버튼) | Amber | `#D8A24A` | 필름/황동 느낌의 포인트 |
| 성공/완료 | Sage | `#7E9B6B` | "적용 완료" 표시 |
| 경고/주의 | Rust | `#C2603C` | 원본 덮어쓰기 등 주의 |

> 라이트 모드도 함께 제공하되, 사진을 다루는 도구는 어두운 배경이 기본값으로 적합합니다.

### 4-3. 타이포그래피 (Typography)

| 용도 | 추천 | 비고 |
| --- | --- | --- |
| 기본 UI 글꼴 | Inter, Pretendard(한글) | 가독성 좋은 산세리프 |
| 숫자·날짜·태그값 | 고정폭(JetBrains Mono 등) | 날짜·수치가 줄 맞아 읽기 쉬움 |
| 제목 포인트 | (선택) 살짝 클래식한 세리프 | 아날로그 감성 한 스푼 |

크기 단계는 단순하게: 제목 20px / 소제목 16px / 본문 14px / 라벨 12px.

### 4-4. 핵심 컴포넌트 (Components)

- **썸네일 카드**: 선택 시 Amber 테두리 + 체크 표시. 적용 완료된 사진은 모서리에 Sage 점.
- **입력 필드**: 라벨은 칸 위에, 같은 값을 여러 장에 적용 중이면 "30장에 적용됨" 안내.
- **날짜 입력기**: 달력 + 직접 타이핑 모두 지원(오래된 연도 입력이 잦으므로 연도 타이핑 필수).
- **자동완성 드롭다운**: 카메라/렌즈/필름은 과거 입력값·프리셋에서 자동완성.
- **주요 버튼(Primary)**: Amber 배경 — "저장", "적용". **보조 버튼**: 외곽선만.
- **상태 토스트**: 우하단에 "30장 저장 완료" 같은 짧은 알림.
- **안전 확인 모달**: 원본을 덮어쓰는 동작에만 Rust 색 경고로 한 번 더 확인.

### 4-5. 디자인 원칙 (톤)

1. **사진이 주인공** — UI는 어둡고 조용하게, 사진과 입력값이 도드라지게.
2. **반복을 없앤다** — 한 통 36컷에 같은 값을 36번 넣게 하지 않는다. 일괄 적용·프리셋·자동완성으로.
3. **되돌릴 수 있다는 안심** — 원본 백업과 명확한 실행취소로 "잘못 누르면 어쩌지" 불안을 없앤다.
4. **연도 입력이 쉬울 것** — 필름은 과거 날짜가 핵심. 오래된 연·월·일을 빠르게 넣을 수 있어야 한다.
