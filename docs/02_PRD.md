# PRD — 필름사진 EXIF 수정 툴

> Product Requirements Document · v1.0
> 작성일: 2026-06-28 · 대상: AI 보조 개발(예: Claude Code 등)으로 구현
> 관련 문서: `01_기획_기능_구조_디자인시스템.md`

---

## 1. 개요

### 1.1 제품 한 줄 정의
필름으로 찍어 스캔한 사진 파일의 EXIF 메타데이터(촬영 날짜, 카메라·렌즈, 필름·현상소 정보)를 비개발자도 쉽고 빠르게, 여러 장을 한 번에 채워 넣는 Windows/Mac 데스크톱 앱.

### 1.2 배경 / 문제
필름 카메라는 EXIF를 기록하지 않는다. 스캔본에는 스캐너의 스캔 일시만 남아 (1) 촬영 날짜가 스캔 날짜로 잘못 들어가고 (2) 카메라·렌즈·필름 정보가 비어 있다. 기존 도구(LensTagger 등)는 Lightroom 플러그인이거나 명령줄 기반이라 비개발자 진입장벽이 높다.

### 1.3 목표 (Goals)
- 한 통(롤) 분량의 스캔본에 촬영 날짜·카메라·렌즈·필름·현상소를 최소 클릭으로 입력
- 롤 전체 공통 정보는 한 번에, 컷별로 다른 날짜는 따로 수정하는 직관적 흐름
- 원본을 안전하게 보존(백업)하여 실수 복구 가능
- 설치형 데스크톱 앱으로 오프라인 동작

### 1.4 비목표 (Non-Goals)
- 사진 보정·색보정·스캔 기능 (이미지 편집은 범위 밖)
- 클라우드 동기화·계정·온라인 서비스 (MVP 기준)
- 모바일 앱 (추후 검토)
- RAW 현상 (DNG의 메타데이터만 수정, 픽셀은 건드리지 않음)

## 2. 대상 사용자 & 핵심 시나리오

**주 사용자**: 필름 사진을 직접 스캔하거나 스캔 맡겨 받은 아카이브를 정리하려는 취미·세미프로 사진가. 코딩 지식 없음.

**핵심 시나리오**
1. 사용자가 한 롤(36컷)을 스캔한 폴더를 앱에 끌어다 놓는다.
2. 오른쪽 "롤 공통" 칸에 `Nikon FM2 / 50mm f1.4 / Portra 400 / OO현상소`를 입력하고 "전체에 적용"을 누른다 → 36컷 모두 반영.
3. 1~12번은 5월 10일, 13~36번은 5월 17일에 찍었다. 1~12번 썸네일을 다중 선택하고 날짜 `1998-05-10`을 입력해 "선택 항목에 적용". 나머지도 동일.
4. "저장"을 누르면 원본이 백업되고 파일의 EXIF가 실제로 수정된다.

## 3. 기능 요구사항 (Functional Requirements)

> 우선순위: **P0 = MVP(1차 출시)**, **P1 = v2**, **P2 = v3**
> 각 요구사항은 수용 기준(AC)으로 완료를 판정한다.

### FR-1 · 사진 불러오기 — P0
파일 다중 선택 또는 폴더/파일 드래그 앤 드롭으로 이미지를 불러온다.
- 지원 포맷: JPEG(.jpg/.jpeg), TIFF(.tif/.tiff), PNG, DNG
- **AC**: 폴더를 드롭하면 하위 이미지가 모두 목록에 뜬다. 미지원 파일은 무시하고 건수를 알린다. 100장 이상도 멈춤 없이 로드된다.

### FR-2 · 썸네일 그리드 & 다중 선택 — P0
불러온 사진을 썸네일 격자로 표시하고 다중 선택을 지원한다.
- 선택 방식: 클릭(단일), Cmd/Ctrl+클릭(개별 토글), Shift+클릭(범위), 전체 선택
- **AC**: 선택된 썸네일은 시각적으로 강조되고, 하단 바에 "선택 N장 / 전체 M장"이 표시된다.

### FR-3 · 미리보기 — P0
선택한 사진(단일)을 가운데 영역에 크게 보여준다.
- **AC**: 썸네일 선택 시 0.5초 내 미리보기가 갱신된다. 다중 선택 시 마지막 선택 컷 또는 대표 1장을 보여준다.

### FR-4 · 롤 공통 정보 입력 & 전체 적용 — P0
카메라 제조사·모델, 렌즈 제조사·모델, 필름 종류, 현상소를 입력하고 불러온 **모든 컷**에 일괄 적용한다.
- 과거 입력값 기반 자동완성 제공
- **AC**: "전체에 적용"을 누르면 현재 불러온 전체 컷의 해당 필드가 입력값으로 채워지고, 적용됨이 시각적으로 표시된다(예: 컷 상태 배지). 빈 칸은 덮어쓰지 않는다.

### FR-5 · 컷별 날짜 입력 & 선택 적용 — P0
촬영 날짜·시간을 입력하고 **선택한 컷**에만 적용한다. 롤 공통 값은 유지된다.
- 날짜 형식 내부 저장: `YYYY:MM:DD HH:MM:SS` (EXIF 표준)
- **AC**: 다중 선택 후 날짜 적용 시 선택분만 바뀌고 비선택분은 그대로다. 잘못된 날짜(존재하지 않는 날짜 등)는 저장 전에 막고 안내한다.

### FR-6 · 날짜 입력 보조 — P0
- **구간 적용**: 선택한 묶음에 같은 날짜 일괄 지정
- **시간 자동 증가**: 같은 날짜의 연속 컷에 1초(또는 사용자가 정한 간격)씩 더해 촬영 순서대로 정렬되게 함
- **느슨한 입력**: 정확한 날짜를 모를 때 연·월만 입력 허용(일/시는 기본값 보충)
- **AC**: 자동 증가 적용 시 각 컷의 `DateTimeOriginal`이 순서대로 단조 증가한다. 연/월만 입력해도 유효한 EXIF 날짜로 저장된다.

### FR-7 · 저장 전 미리보기(변경 요약) — P0
저장 전에 "어떤 컷의 어떤 태그가 무엇으로 바뀌는지" 요약을 보여준다.
- **AC**: 저장 버튼을 누르면 변경 대상 건수와 주요 변경 내용을 확인하는 단계를 거친다.

### FR-8 · 원본 보존(백업/사본) — P0
저장 시 원본을 보존하는 방식을 **저장 직전에 선택**한다.
- 모드 A · 제자리 수정 + 백업(기본): 원본을 **같은 폴더의 `original/` 하위 폴더에 같은 이름으로 복사**해 둔 뒤 파일을 직접 고친다. (이미지로 그대로 열리는 백업 — ExifTool의 `_original` 접미사 방식은 쓰지 않는다.) 폴더 스캔 시 `original/`은 목록에서 제외한다.
- 모드 B · 사본으로 저장: 원본은 전혀 건드리지 않고, **지정한 별도 출력 폴더에 원본 파일명 그대로** 수정본을 새로 만든다(ExifTool `-o` 사용). 출력 폴더에 같은 이름이 이미 있으면 덮어쓰기 전에 확인한다.
- **AC**: 저장 후에도 원본을 되돌릴 방법이 항상 존재한다(모드 A는 `original/` 폴더 백업, 모드 B는 원본 무수정). 저장 직전에 저장 모드(및 모드 B의 출력 폴더)를 선택할 수 있다.

### FR-9 · 변경 저장(EXIF 쓰기) — P0
실제 파일에 EXIF를 기록한다. 내부 엔진으로 ExifTool 사용.
- 저장 대상 태그: 아래 §4 데이터 모델 참조
- **AC**: 저장 후 ExifTool로 다시 읽으면 입력값이 그대로 확인된다. 다른 EXIF 뷰어(예: macOS 정보 창)에서도 촬영일이 의도대로 보인다. 파일 손상이 없다.

### FR-10 · 필름·현상소 태그 매핑 — P0
- **필름 종류 → `ImageDescription`(EXIF)** 에 기록하고, 호환을 위해 `XMP-dc:Description` 에도 병행 기록한다. (필름 이름 전용 표준 태그가 없어, 필름 사진가들이 널리 쓰는 설명 필드를 사용)
- **현상소 → `UserComment`** 에 `DevLab: ...` 형태로 기록한다(표준 태그 없음).
- **필름 감도(ISO, 박스 감도) → `ISOSpeedRatings`**(=EXIF `ISO`), **노출지수(EI, 실제 촬영 감도) → `ExposureIndex`**. 정상 노출이면 둘이 같고, 증감(푸시/풀) 촬영이면 다르다. 둘 다 롤 공통 입력.
- **AC**: 화면에는 "필름", "현상소" 전용 칸으로 노출되지만 저장 시 위 태그에 기록되고, 다시 불러오면 각 칸에 복원된다(필름은 ImageDescription 우선, 레거시 UserComment의 Film도 인식).

### FR-11 · 프리셋(조합 저장) — P1
자주 쓰는 카메라+렌즈+필름+현상소 조합을 이름 붙여 저장/적용.
- **AC**: 프리셋 선택 한 번으로 롤 공통 칸이 모두 채워진다.

### FR-12 · 롤(세션) 관리 — P1
불러온 사진 묶음을 "롤"로 저장하고 나중에 이어서 편집.
- **AC**: 앱을 껐다 켜도 진행 중이던 롤의 입력 상태가 복원된다.

### FR-13 · 실행 취소/다시 실행 — P1
입력·적용 동작에 대한 Undo/Redo.
- **AC**: 직전 적용을 한 번의 동작으로 되돌릴 수 있다.

### FR-14 · CSV 가져오기/내보내기 — P1
컷별 정보를 표로 입력/백업.
- **AC**: 파일명-날짜 매핑 CSV를 불러오면 해당 컷에 날짜가 일괄 입력된다.

### FR-15 · GPS 위치 지정 — P2
지도 또는 좌표 입력으로 `GPSLatitude/GPSLongitude` 기록.

### FR-16 · 필름 데이터베이스 자동완성 — P2
주요 필름 목록 내장 자동완성.

### FR-17 · 스캐너 메타데이터 정리 — P1
필름 스캔본에 스캐너/스캔 소프트웨어가 남긴 메타데이터를 **선별 삭제**한다.
- 삭제 대상(스캐너 출처): `Software`, `ProcessingSoftware`, `HostComputer`, 스캐너 메이커노트(`MakerNotes`), 스캐너 `Make`/`Model`(카메라 정보를 입력하면 FR-4에서 어차피 덮어써짐), 스캐너가 기록한 IPTC/XMP 항목. 별도 `.xmp` 사이드카 파일이 있으면 함께 정리.
- 보존: 스캔 일시는 지우지 않고 **디지털화 날짜**(`CreateDate`/`DateTimeDigitized`)로 남겨 촬영일(`DateTimeOriginal`)과 구분한다.
- 안전: 전체 삭제(`-all=`)는 쓰지 않는다(정당한 EXIF 손실·불완전 삭제·사이드카 누락 위험). 선별 목록만 지우고, 저장 전에 "지워질 항목"을 요약해 확인받는다. 삭제도 FR-8 백업·검증 파이프라인을 그대로 통과한다.
- **AC**: "스캐너 정보 지우기"를 켜고 저장하면 위 스캐너 태그가 제거되고, 촬영일·디지털화 날짜와 사용자가 입력한 카메라·렌즈·필름 정보는 보존된다. 저장 후 다른 뷰어에서 스캐너 흔적이 보이지 않는다.

## 4. 데이터 모델 — EXIF 필드 매핑

입력값은 두 층으로 관리한다. **롤 공통**은 전체 컷 동일, **컷별**은 컷마다 다름.

### 롤 공통
| 입력칸 | EXIF 태그 | 형식/예시 |
| --- | --- | --- |
| 카메라 제조사 | `Make` | 문자열 · Nikon |
| 카메라 모델 | `Model` | 문자열 · FM2 |
| 렌즈 제조사 | `LensMake` | 문자열 · Nikon |
| 렌즈 모델 | `LensModel` | 문자열 · 50mm f/1.4 |
| 필름 종류 | `ImageDescription`(+XMP 설명) | `Portra 400` |
| 현상소 | `UserComment` | `DevLab: OO현상소` |

### 컷별
| 입력칸 | EXIF 태그 | 형식/예시 |
| --- | --- | --- |
| 촬영 날짜·시간 | `DateTimeOriginal` | `YYYY:MM:DD HH:MM:SS` · `1998:05:10 12:00:00` |
| (보조) 디지털화 날짜 | `CreateDate` / `DateTimeDigitized` | 스캔 일시 보존용(선택) |

### 확장(P1~P2)
초점거리 `FocalLength`, 조리개 `FNumber`, 셔터스피드 `ExposureTime`, 감도 `ISOSpeedRatings`/`ExposureIndex`, 위치 `GPSLatitude`/`GPSLongitude`.

> 필름은 `ImageDescription`(+XMP 설명)에, 현상소는 `UserComment`에 `DevLab: ...` 형태로 기록한다. 불러올 때 필름은 ImageDescription에서, 현상소는 UserComment에서 복원한다(규칙은 Rust 한 곳).
>
> **스캐너 정리(FR-17)**: 위 매핑과 별개로, 스캐너/스캔SW가 남긴 태그(`Software`·`ProcessingSoftware`·`HostComputer`·`MakerNotes`·스캐너 `Make`/`Model` 등)는 선별 삭제 대상이다. 스캔 일시는 디지털화 날짜(`CreateDate`/`DateTimeDigitized`)로 **보존**한다.

## 5. 기술 요구사항

| 영역 | 선택 | 비고 |
| --- | --- | --- |
| 앱 프레임워크 | **Tauri v2** | 경량·보안 우수. 대안: Electron |
| 프런트엔드 | React + TypeScript + Tailwind CSS | AI 도구 지원·자료 풍부 |
| 백엔드(네이티브) | Rust (Tauri) | 파일 접근·ExifTool 호출 |
| EXIF 엔진 | **ExifTool**(사이드카 바이너리 번들) | 검증된 표준. Rust에서 `Command`로 호출 |
| 상태 저장 | 로컬 파일/SQLite(선택) | 프리셋·롤 세션 보관 |

**핵심 통합 메모**
- ExifTool 바이너리를 앱에 동봉(sidecar)하여 사용자가 별도 설치하지 않게 한다.
- 쓰기는 한 컷씩 또는 배치로 호출하되, 대량 처리 시 진행률을 표시한다.
- 모든 파일 작업은 백업 정책(FR-8)을 통과한 뒤 수행한다.

## 6. UX / 화면 요구사항

### 6.1 레이아웃 — 3분할
왼쪽 썸네일 그리드 / 가운데 미리보기 / 오른쪽 편집 폼. 하단 상태바(선택·전체 건수, 프리셋, 저장).

### 6.2 편집 폼 — 2구역 분리
오른쪽 폼은 **[롤 공통]**(전체에 적용 버튼)과 **[컷별 — 선택분]**(선택 항목에 적용 버튼)으로 나눈다. 화면 구조가 2단계 워크플로를 그대로 안내한다.

### 6.3 디자인 시스템
`01_기획_기능_구조_디자인시스템.md` §4를 따른다. 다크 필름톤 팔레트(Ink/Amber/Sage/Rust), Inter/Pretendard, 날짜·태그값은 고정폭. 원본 덮어쓰기 등 위험 동작에만 경고 모달.

### 6.4 상태 표시
각 컷에 적용 상태 배지(미적용 / 공통 적용됨 / 날짜 적용됨 / 저장됨). 작업 결과는 우하단 토스트로.

## 7. 비기능 요구사항

- **안전성**: 저장 전 원본 백업 보장. 저장 실패 시 원본 무손상.
- **성능**: 200장 로드 및 일괄 저장이 합리적 시간 내(목표: 200장 저장 30초 이내) 완료, 진행률 표시.
- **호환성**: Windows 10+ / macOS 12+. 저장 결과가 표준 EXIF 뷰어와 호환.
- **오프라인**: 네트워크 없이 전 기능 동작.
- **현지화**: 한국어 우선, 추후 영어(P2).
- **데이터 보존**: 기존 EXIF 중 사용자가 건드리지 않은 태그는 보존.

## 9. 수용 기준 (전체 MVP 합격선)

다음이 모두 참이면 MVP 완료로 본다.
1. 폴더 드롭 → 썸네일 표시 → 롤 공통 입력·전체 적용 → 구간별 날짜 입력 → 저장까지 막힘 없이 수행된다.
2. 저장 후 다른 EXIF 뷰어에서 촬영일·카메라·렌즈가 의도대로 보인다.
3. 필름·현상소가 `UserComment`에 기록되고 앱에서 다시 읽으면 칸에 복원된다.
4. 모든 저장은 되돌릴 수 있는 원본 백업을 남긴다.
5. 200장 규모에서 충돌·데이터 손상이 없다.
