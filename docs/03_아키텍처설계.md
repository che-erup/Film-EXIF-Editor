# 아키텍처 설계 — 필름사진 EXIF 수정 툴

> Architecture Design Document · v1.0
> 작성일: 2026-06-28 · 관련: `02_PRD.md`
> 설계 목표: **효율적(빠름) · 안정적(데이터 무손실) · 단순(AI로 구현·유지 쉬움)**

---

## 1. 설계 원칙 (왜 이렇게 짓는가)

이 앱은 "남의 사진 파일을 직접 고쳐 쓰는" 도구다. 따라서 두 가지가 최우선이다.

1. **절대 데이터를 잃지 않는다.** 모든 저장은 백업 → 검증 → 확인의 안전 파이프라인을 통과한다.
2. **수백 장을 빠르게 처리한다.** EXIF 엔진을 매번 새로 띄우지 않고, 한 번 띄워 계속 쓴다.

여기서 파생되는 4가지 원칙:

- **메모리 우선 편집(Edit-in-memory)** — 입력·적용은 전부 메모리에서만 일어난다. 디스크 파일은 "저장"을 누를 때 **단 한 번** 건드린다. 그 전까지 원본은 손대지 않으므로 안전하다.
- **단일 책임 경계(Trust boundary)** — 파일을 실제로 읽고 쓰는 일은 전부 네이티브(Rust) 쪽 한 곳에서만 한다. 화면(React)은 파일을 직접 만지지 않는다.
- **엔진 상주(Long-lived engine)** — ExifTool을 `-stay_open` 모드로 한 번만 띄우고 명령을 계속 흘려보낸다. (개별 호출 대비 최대 약 10배 빠름)
- **부분 실패 격리** — 200장 중 3장이 실패해도 나머지 197장은 정상 저장되고, 실패분만 따로 보고·재시도한다.

## 2. 전체 구조 (4계층)

```
┌──────────────────────────────────────────────────────────────┐

                                 ▼
┌──────────────────────────────────────────────────────────────┐
│  ② 애플리케이션 코어 — Rust (Tauri 백엔드)                     │

               ▼                              ▼
┌──────────────────────────┐   ┌──────────────────────────────┐
│  ③ ExifTool 서비스        │   │  ③ 썸네일/이미지 서비스       │
│   · stay_open 상주 프로세스│   │   · 미리보기·썸네일 디코딩    │
```

**계층을 나누는 이유**: 화면(①)과 파일 작업(②③)을 분리하면, 화면이 멈추거나 바뀌어도 파일 안전 로직은 영향을 받지 않는다. 또 파일 로직이 한 곳(Rust)에 모여 있어 테스트와 검증이 쉽다.

## 3. 핵심 설계 결정 3가지

### 결정 1 · ExifTool을 "상주 프로세스"로 (효율)

가장 흔한 실수는 사진 한 장마다 `exiftool` 명령을 새로 실행하는 것이다. ExifTool은 Perl 인터프리터와 거대한 라이브러리를 매번 로드하므로, 이 방식은 200장에서 수십 초~수 분이 걸린다.

대신 **`-stay_open True -@ -` 모드**로 프로세스를 앱 시작 시 한 번 띄우고, 표준입력 파이프로 명령을 계속 보낸다. 명령 사이에 `-execute`를 넣어 구분하고, 결과는 JSON(`-j`)으로 받는다. 이 방식은 한 번 로드한 엔진을 재사용하므로 **최대 약 10배 빠르다.** ([ExifTool 성능 문서](https://deepwiki.com/exiftool/exiftool/10.1-performance-optimization), [stay_open 사용법](https://exiftool.org/forum/index.php?topic=3139.0))

Rust 쪽에 `ExifToolService`를 두어 이 프로세스의 수명(시작·요청 직렬화·종료)을 책임진다. 프로세스는 하나뿐이므로 요청은 큐로 순차 처리한다.

```rust
// 개념 코드 (의사)
struct ExifToolService { child: Child, stdin: ChildStdin, stdout: BufReader<ChildStdout> }

impl ExifToolService {
    fn start(sidecar_path) -> Self { /* exiftool -stay_open True -@ - 실행 */ }
    fn execute(&mut self, args: &[String]) -> Result<String> {
        // args를 한 줄씩 쓰고 "-execute\n" → {ready} 마커까지 stdout 읽기
    }
    fn shutdown(&mut self) { /* "-stay_open\nFalse\n" 전송 */ }
}
```

### 결정 2 · 메모리 우선 편집 모델 (안정 + 효율)

편집 세션 전체를 메모리의 데이터 구조로 들고 있는다. "전체에 적용", "선택분에 날짜 적용" 같은 동작은 **메모리 값만 바꾼다.** 디스크 파일은 "저장"을 누를 때만 쓴다.

- 장점(안정): 저장 전에는 원본이 그대로다. 실수해도 닫으면 그만이다.
- 장점(효율): 100번 입력을 고쳐도 디스크 쓰기는 0번. 저장 때 한 번에 처리한다.
- Undo/Redo도 메모리 상태 스냅샷으로 간단히 구현된다.

각 컷은 `원본 태그(originalTags)` 와 `편집 의도(pendingEdits)` 를 함께 들고, 화면에는 둘을 합친 결과를 보여준다. 저장 시에는 **pendingEdits(바뀐 것)만** 파일에 기록한다.

### 결정 3 · 안전 저장 파이프라인 (안정)

"저장"은 단순히 쓰는 게 아니라 4단계를 거친다. 각 파일은 독립적으로 처리되어 한 장이 실패해도 다른 장에 영향이 없다.

```
컷마다:
 1) 검증(Validate)   날짜 형식·유효성, 인코딩 점검 → 통과 못하면 건너뛰고 표시
 2) 백업(Backup)     ExifTool 기본 _original 백업 유지 + (옵션) 별도 폴더 사본
 3) 쓰기(Write)      바뀐 태그만 stay_open으로 기록
 4) 검증읽기(Verify) 쓴 직후 다시 읽어 의도값과 일치 확인
결과: SaveResult { ok / 실패사유 } 를 컷별로 수집 → 요약 보고
```

핵심 안전장치:
- **원본은 항상 되돌릴 수 있다** — ExifTool은 기본적으로 원본을 `파일명_original`로 남긴다. 이 동작을 끄지 않는다. 추가로 사용자가 원하면 별도 백업 폴더에도 복사한다.
- **건드리지 않은 태그는 보존** — 바뀐 태그만 쓰므로 기존 EXIF가 날아가지 않는다.
- **검증읽기 실패 시 경고** — 쓴 값이 다시 읽었을 때 다르면 그 컷을 실패로 표시한다.

**저장 모드(FR-8)**: 저장 직전에 두 방식 중 선택한다 — (A) 제자리 수정 + `_original` 백업, (B) 사본 저장(원본 무수정, 지정한 별도 출력 폴더에 원본과 같은 이름; ExifTool `-o` 로 새 파일 생성). 두 모드 모두 위 4단계(검증 · 백업/사본 · 쓰기 · 검증읽기)를 동일하게 거친다. 모드 B에서는 ②백업 대신 ‘사본 생성’이 되고 원본은 손대지 않는다.

## 4. 데이터 모델

UI(TS)와 코어(Rust)가 같은 모양을 공유한다(타입 동기화).

```ts
// 롤 공통 정보 — 전체 컷 동일
interface RollCommon {
  make?: string;          // EXIF Make
  model?: string;         // EXIF Model
  lensMake?: string;      // EXIF LensMake
  lensModel?: string;     // EXIF LensModel
  filmStock?: string;     // → UserComment "Film: ..."
  devLab?: string;        // → UserComment "DevLab: ..."
}

// 컷별 정보 — 사진마다 다름
interface FrameEdits {
  dateTimeOriginal?: string;  // "YYYY:MM:DD HH:MM:SS"
}

// 한 장
interface Frame {
  id: string;
  path: string;
  fileName: string;
  format: "jpeg" | "tiff" | "png" | "dng";
  originalTags: Record<string, string>;  // 불러올 때 읽은 기존값(읽기전용)
  pendingCommon: Partial<RollCommon>;     // 이 컷에 적용된 공통값
  pendingEdits: FrameEdits;               // 이 컷의 개별 편집
  status: "loaded" | "commonApplied" | "dateApplied" | "saved" | "error";
}

// 편집 세션(=롤)
interface EditSession {
  id: string;
  frames: Frame[];
  rollCommon: RollCommon;   // 입력 폼의 현재 공통값
}
```

**UserComment 직렬화 규칙** (필름·현상소는 표준 태그가 없으므로):
- 쓰기: `Film: {filmStock}; DevLab: {devLab}` 형태로 합쳐 `UserComment`에 기록, 동일 내용을 XMP에도 병행 기록(호환성).
- 읽기: `키: 값; 키: 값` 규칙으로 파싱해 각 칸 복원.
- 규칙을 한 곳(Rust의 직렬화 모듈)에만 두어 쓰기/읽기 불일치를 방지한다.

**스캐너 메타데이터 정리 규칙** (FR-17):
- 스캔본에는 스캐너/스캔SW가 남긴 태그(`Software`, `ProcessingSoftware`, `HostComputer`, 스캐너 `MakerNotes`, 스캐너 `Make`/`Model`, 스캐너가 쓴 IPTC/XMP)가 들어 있다. 이들만 **선별 삭제**한다(전체 `-all=` 금지 — 정당한 EXIF 손실·불완전 삭제·`.xmp` 사이드카 누락 위험).
- 스캔 일시는 삭제하지 않고 디지털화 날짜(`CreateDate`/`DateTimeDigitized`)로 보존해 촬영일과 구분한다.
- 삭제 대상 목록도 직렬화 모듈과 같은 한 곳(Rust)에 두고, 삭제는 §3 결정3의 백업→쓰기→검증 파이프라인을 그대로 통과한다.

## 5. 데이터 흐름 (주요 동작 4가지)

**A. 불러오기**
```
사용자 폴더 드롭 → [UI] load_images(paths) 호출
 → [코어] 이미지 파일 필터링 → [ExifTool] 각 파일 기존 태그 읽기(-j 배치)
 → Frame[] 생성 → [UI] 썸네일 그리드 렌더 (썸네일은 비동기 별도 로드)
```

**B. 롤 공통 적용** (디스크 안 건드림)
```
사용자 "전체에 적용" → [UI] 세션의 모든 frame.pendingCommon 갱신
 → 상태 배지 "공통 적용됨" → (저장 전까지 메모리만 변경)
```

**C. 컷별 날짜 적용** (디스크 안 건드림)
```
다중 선택 + 날짜 입력 + "선택 항목에 적용"
 → 선택된 frame들의 pendingEdits.dateTimeOriginal 갱신
 → (자동증가 옵션 시 컷 순서대로 +간격) → 상태 "날짜 적용됨"
```

**D. 저장** (유일하게 디스크를 쓰는 동작)
```
"저장" → [코어] save_changes(session) 비동기 작업 시작
 → 컷마다 §3 결정3의 4단계 파이프라인 실행
 → 진행률 event를 [UI]로 emit (N/200)
 → 완료 시 SaveResult[] 반환 → [UI] "197 성공 / 3 실패" 요약 + 실패분 재시도 버튼
```

## 6. 동시성 / 성능 설계

- **두 개의 독립 레인**:
  - *메타데이터 레인* — ExifTool 상주 프로세스 하나를 통해 **순차** 처리(프로세스가 하나이므로). stay_open 덕에 순차여도 충분히 빠름.
  - *썸네일 레인* — 이미지 디코딩은 CPU 작업이라 **병렬**(Rust `image` 크레이트 + 스레드풀). 메타데이터 작업을 막지 않는다.
- **가상화 그리드** — 썸네일은 화면에 보이는 만큼만 렌더(가상 스크롤). 수백 장도 부드럽게.
- **썸네일 캐시** — 가능하면 파일에 내장된 EXIF 썸네일을 우선 사용, 없으면 축소본 생성 후 앱 캐시에 저장.
- **진행률·취소** — 저장은 백그라운드 작업으로 진행률을 이벤트로 보내고, 컷 사이마다 취소 플래그를 확인해 중단 가능.

## 7. 안정성 / 에러 처리

| 상황 | 처리 |
| --- | --- |
| 저장 중 일부 파일 실패(권한·손상 등) | 해당 컷만 error로 표시, 나머지는 계속. 실패 목록·사유 제공, 재시도 버튼 |
| 쓰기 후 검증읽기 불일치 | 그 컷 error 처리, 원본(_original)으로 복구 안내 |
| 앱 비정상 종료 | 편집 의도(세션 메타데이터)를 앱 데이터 폴더에 주기적 자동저장 → 재시작 시 복원. (파일 자체는 저장 전이라 무손상) |
| ExifTool 프로세스 죽음 | 서비스가 감지 후 자동 재시작, 진행 중 작업은 실패 보고 |
| 잘못된 날짜·인코딩 | 저장 전 검증에서 차단, 입력 시점 인라인 경고 |
| 위험 동작(원본 직접 덮어쓰기 모드) | 경고 모달로 명시적 확인 |

([Tauri 사이드카 문서](https://v2.tauri.app/develop/sidecar/), [Shell 플러그인](https://v2.tauri.app/plugin/shell/))

## 9. 테스트 전략

- **단위(Rust)**: UserComment 직렬화↔파싱 왕복, 날짜 검증, 저장 파이프라인 분기(성공/실패/검증불일치).
- **통합**: 실제 샘플 파일(JPEG/TIFF/DNG 각각)로 쓰기→읽기 왕복이 의도대로인지. 다른 뷰어 호환 확인.
- **안전성 회귀**: 저장 후 `_original` 존재 및 복구 가능 여부, 비대상 태그 보존 여부.
- **성능**: 200장 저장 시간 측정(목표 30초 이내), stay_open on/off 비교로 회귀 감시.
- **부분 실패**: 일부러 읽기전용·손상 파일을 섞어 격리·재시도가 동작하는지.

---

## 10. 구현 순서 (얇은 수직 슬라이스 우선)

기능을 가로로 쌓지 말고, **한 장을 끝까지 저장**하는 가장 얇은 경로부터 세로로 관통한다.

1. **M0 — 관통 슬라이스**: Tauri+React 뼈대 → exiftool 사이드카 stay_open 연결 → 사진 1장 불러오기 → `DateTimeOriginal` 바꿔 안전 저장(백업·검증읽기 포함) 성공. *여기서 가장 위험한 통합(사이드카·서명·stay_open)을 먼저 제거한다.*
2. **M1 — MVP 폭 확장**: 다중 불러오기·썸네일 그리드 → 롤 공통 적용 → 다중 선택 날짜 적용 → 배치 저장·진행률 → UserComment 매핑.
3. **M2 — 편의**: 프리셋·롤 세션 자동복원·Undo/Redo·CSV.
4. **M3 — 확장**: GPS·필름 DB.

## 11. 한눈에 보는 "효율 vs 안정" 요약

| 목표 | 어떻게 달성하나 |

| **빠름** | ExifTool stay_open 상주(최대 10배), 메모리 우선 편집(디스크 쓰기 최소화), 썸네일 병렬·가상화 |
| **안 깨짐** | 저장 전 원본 무접촉, `_original` 백업 유지, 바뀐 태그만 기록, 쓰기 후 검증읽기 |
| **부분 실패 견딤** | 컷별 독립 처리, 실패 격리·재시도, 세션 자동복원 |
